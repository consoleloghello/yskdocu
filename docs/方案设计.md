 ## 一、技术方案概述
 
 总体思路：用 Python 解析 docx 题库文件，输出结构化 JSON；前端为纯静态 SPA（HTML + CSS + Vanilla JS），无后端依赖，直接部署 GitHub Pages。用户选择版本（外操/内操）后，按章节浏览题目卡片，支持关键字搜索。
 
 ## 二、技术选型
 
 | 类别 | 选择 | 理由 |
 |------|------|------|
 | 文档解析 | Python + python-docx | 已安装，能可靠读取 docx 段落结构与文本内容 |
 | 前端框架 | Vanilla JS（无框架） | 单页应用、无路由需求；零构建、零依赖，直接部署 |
 | 样式方案 | CSS3（Flexbox/Grid） | 原生响应式布局，轻量无依赖 |
 | 数据存储 | 静态 JSON 文件 + localStorage | JSON 随页面加载；答题状态/错题本存 localStorage |
 | 部署 | GitHub Pages | 纯静态文件，零运维成本 |
 | 图标 | Unicode/Emoji | 无需引入图标库，减少体积 |
 
 ## 三、架构设计
 
 ### 数据流
 
 ```
 .docx 文件 ──> Python 解析脚本 ──> JSON 题库文件 ──> SPA 加载 ──> 用户浏览
                                                      │
                                                      └──> localStorage (状态持久化)
 ```
 
 ### 系统分层
 
 | 层 | 职责 | 文件 |
 |------|------|------|
 | 数据层 | docx 解析、JSON 输出 | `scripts/parse_neicao.py` / `parse_waicao.py` |

 | 表现层 | UI 渲染、交互逻辑 | `index.html` + `css/style.css` + `js/app.js` |
 | 持久层 | 状态存取（localStorage） | `js/app.js` 中的 StateManager 模块 |

 ## 六、关键设计决策
 
 | 决策点 | 选择 | 理由 |
 |--------|------|------|
 | 题型筛选 vs 合并展示 | 选择题型标签切换（不在章节级过滤） | 用户可先选章节，再筛选题型，交互清晰 |
 | 答案展示方式 | 按钮「显示答案」点击揭示，而非自动展示 | 适合记忆型刷题，先思考再看答案 |
 | docx 解析策略 | 纯文本模式（忽略样式），基于正则模式匹配 | docx 全为 Normal 样式，无法靠样式区分层级，只能靠文本模式检测 |
 | 内操版/外操版数据 | 两个独立 JSON 文件，前端切换加载 | 结构独立，互不干扰，减少出错 |
 | 搜索范围 | 全文搜索（题目 + 答案 + 章节名） | 简答题答案中有关键操作步骤，需要被搜索到 |
 
 ## 七、docx 解析策略
 
 解析步骤：
 
 1. **读取段落**：用 python-docx 读取所有段落文本，过滤空行
 2. **分割章节**：检测段落文本是否为章节标题（如"给水加压泵站"、"罐区" — 文本前无数字编号前缀的纯文字段落，且后跟"一、选择题"）
 3. **分割题型**：检测 "一、选择题"、"二、填空题"、"三、判断题"、"四、简答题"、"五、实操分析题" 段落
 4. **解析题目**：
    - **选择题**：题目行（包含（答案标记））+ 后续 A./B./C./D. 行
    - **判断题**：题目行（包含（√）/（×）标记）
    - **填空题**：独立题干行（无特殊标记，即普通描述性语句）
    - **简答题**：题干行 + 后续"答案："/"答："开头的回答行
    - **实操分析题**：题干行 + 后续"答案："开头的回答行
 5. **特殊情况处理**：
    - 判断题可能有 "答案：×" 后缀
    - 多选题选项可能在一行（如 "A. xxx B. xxx C. xxx D. xxx"）
    - 简答题答案可能跨多段落
