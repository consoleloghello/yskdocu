## 2026-06-28

## 一、需求背景

公用工程题库 SPA 当前为纯前端应用，所有状态存储在 localStorage 中。存在以下局限：
- 错题本仅限单设备，换设备后丢失
- 无法收集用户对题目质量的反馈
- 缺少答题统计，无法评估学习效果
- 无跨设备同步能力

本次迭代新增 Supabase 后端，在不破坏现有匿名体验的前提下，为登录用户提供云同步功能。

## 二、技术选型

| 类别 | 选择 | 理由 |
|------|------|------|
| 后端数据库 | Supabase (PostgreSQL) | 免费 500MB，自带 REST API，无需写后端代码 |
| 认证 | Supabase Auth (email/password) | 内置邮箱登录，支持白名单，RLS 集成 |
| 前端 SDK | @supabase/supabase-js (CDN) | UMD 格式，直接 script 标签引入，零构建 |
| 统计图表 | Chart.js (CDN) | 轻量（60KB gzipped），饼图/柱状图开箱即用 |
| 前端部署 | GitHub Pages（不变） | 纯静态文件，零运维成本 |
| 后端部署 | Supabase 托管（不变） | 无需运维，免费额度充足 |

**为什么不写后端代码？**

Supabase 提供：
- 自动生成的 REST API（基于 PostgreSQL 表结构）
- Row Level Security 保证用户只能访问自己的数据
- 前端 JS SDK 直接操作数据库

前端只需调用 `supabase.from('wrong_questions').insert({...})` 即可写入数据库，无需编写任何服务端接口。

### 3.2 数据流

```
【匿名模式】（现有逻辑不变）
  flatQs[] → renderCards() → 用户操作 → localStorage

【登录模式】
  flatQs[] → renderCards() → 用户操作 → localStorage（缓存） + Supabase（主存储）
                                       │
                                       ├─ 做错题 → wrong_questions INSERT
                                       ├─ 选答案 → answer_history INSERT
                                       ├─ 写笔记 → question_notes UPSERT
                                       └─ 报错   → question_reports INSERT

【登录初始化】
  Supabase session 恢复 → 拉取云端错题本 → 合并到本地 wrongBook
                        → 拉取云端笔记   → 合并到本地 notes
```

### 3.3 错误处理策略

| 场景 | 处理 |
|------|------|
| 网络断开 | localStorage 正常读写，云端操作静默失败，下次同步时重试 |
| 登录过期 | Supabase SDK 自动刷新 token，失败则提示重新登录 |
| 云端写入失败 | 不阻塞 UI，控制台记录错误，用户无感知 |
| 并发冲突 | 笔记/错题使用 UPSERT，以最后写入为准 |

## 四、数据库设计

### 4.1 表结构

```sql
-- 用户资料表，在用户注册时通过触发器自动创建
CREATE TABLE public.profiles (
  id          uuid PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  email       text,
  nickname    text,
  created_at  timestamptz DEFAULT now()
);

-- 错题本：记录用户做错的题目
CREATE TABLE public.wrong_questions (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     uuid REFERENCES profiles(id) ON DELETE CASCADE NOT NULL,
  version     text NOT NULL,           -- '外操版' 或 '内操版'
  question_id text NOT NULL,           -- 题目 _id，如 '火炬_选择题_0'
  chapter     text,                    -- 章节名（冗余，方便统计）
  type        text,                    -- 题型（冗余）
  created_at  timestamptz DEFAULT now(),
  UNIQUE(user_id, version, question_id)
);

-- 答题历史：记录每次答题的对错
CREATE TABLE public.answer_history (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     uuid REFERENCES profiles(id) ON DELETE CASCADE NOT NULL,
  version     text NOT NULL,
  question_id text NOT NULL,
  chapter     text,
  type        text,
  is_correct  boolean NOT NULL,
  created_at  timestamptz DEFAULT now()
);

-- 题目报错：用户提交的纠错反馈
CREATE TABLE public.question_reports (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     uuid REFERENCES profiles(id) ON DELETE CASCADE NOT NULL,
  version     text NOT NULL,
  question_id text NOT NULL,
  reason      text NOT NULL,           -- 报错原因，如 '答案有误' '题目描述不清' '选项缺失'
  detail      text,                    -- 详细说明
  status      text DEFAULT 'pending',  -- pending | reviewed | fixed
  created_at  timestamptz DEFAULT now()
);

-- 题目笔记：用户的私人笔记
CREATE TABLE public.question_notes (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     uuid REFERENCES profiles(id) ON DELETE CASCADE NOT NULL,
  version     text NOT NULL,
  question_id text NOT NULL,
  content     text NOT NULL,
  updated_at  timestamptz DEFAULT now(),
  UNIQUE(user_id, version, question_id)
);
```

### 4.2 索引

```sql
-- 加速用户按版本的错题查询
CREATE INDEX idx_wrong_questions_user_version
  ON wrong_questions(user_id, version);

-- 加速统计查询
CREATE INDEX idx_answer_history_user_version
  ON answer_history(user_id, version);

CREATE INDEX idx_answer_history_chapter
  ON answer_history(user_id, version, chapter);

-- 加速笔记查询
CREATE INDEX idx_question_notes_user_version
  ON question_notes(user_id, version);
```

### 4.3 RLS 策略

```sql
-- 所有表启用 RLS
ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;
ALTER TABLE wrong_questions ENABLE ROW LEVEL SECURITY;
ALTER TABLE answer_history ENABLE ROW LEVEL SECURITY;
ALTER TABLE question_reports ENABLE ROW LEVEL SECURITY;
ALTER TABLE question_notes ENABLE ROW LEVEL SECURITY;

-- 用户只能读写自己的数据
CREATE POLICY "Users can manage own profile"
  ON profiles FOR ALL USING (auth.uid() = id);

CREATE POLICY "Users can manage own wrong questions"
  ON wrong_questions FOR ALL USING (auth.uid() = user_id);

CREATE POLICY "Users can manage own answer history"
  ON answer_history FOR ALL USING (auth.uid() = user_id);

CREATE POLICY "Users can manage own reports"
  ON question_reports FOR ALL USING (auth.uid() = user_id);

CREATE POLICY "Users can manage own notes"
  ON question_notes FOR ALL USING (auth.uid() = user_id);
```

### 4.4 自动创建 profile 触发器

```sql
-- 新用户注册时自动在 profiles 表创建记录
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS trigger AS $$
BEGIN
  INSERT INTO public.profiles (id, email, nickname)
  VALUES (
    NEW.id,
    NEW.email,
    COALESCE(NEW.raw_user_meta_data ->> 'nickname', split_part(NEW.email, '@', 1))
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE OR REPLACE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();
```

## 六、认证流程

### 6.1 白名单机制

Supabase 配置：
1. 关闭公开注册（Dashboard → Authentication → Settings → 关闭 "Enable Sign Up"）
2. 管理员在 Dashboard → Authentication → Users 手动创建用户
3. 或通过 SQL 直接插入 `auth.users`（需用 Supabase 提供的 `create_user` 函数）

### 6.2 登录流程

```
用户点击「登录」
  → 弹出登录弹窗
  → 输入邮箱 + 密码
  → SupabaseAuth.signIn(email, password)
  → Supabase Auth 验证
  → 成功：关闭弹窗，显示用户昵称，触发云端同步
  → 失败：显示错误提示

页面刷新
  → SupabaseAuth.getSession()
  → 有 session：恢复登录状态，拉取云端数据
  → 无 session：保持匿名模式
```

## 七、部署与配置

### 7.1 Supabase 项目创建

1. 访问 https://supabase.com，注册/登录
2. 创建新项目，选择最近的区域（建议 Singapore 或 Tokyo）
3. 设置数据库密码（妥善保存）
4. 等待项目初始化（约 2 分钟）

在 Supabase Dashboard → SQL Editor 中执行 `scripts/init_supabase.sql`（包含建表 + 索引 + RLS + 触发器）。

### 7.3 认证配置

1. Dashboard → Authentication → Settings：
   - 关闭 "Enable Sign Up"（禁止公开注册）
   - 关闭 "Enable Anonymous Sign-Ins"
2. Dashboard → Authentication → Users → "Add User" 手动创建白名单用户

### 7.4 前端配置

修改 `js/supabase.js` 中的常量：

## 八、安全性考量

| 方面 | 措施 |
|------|------|
| 数据传输 | Supabase JS SDK 默认 HTTPS |
| 数据隔离 | RLS 策略保证用户只能访问自己的数据，`anon_key` 无法绕过 |
| 密码安全 | Supabase Auth 使用 bcrypt 哈希存储密码 |
| API Key 暴露 | `anon_key` 是公开的（设计如此），安全依赖 RLS 而非密钥保密 |
| XSS | 保持现有 `esc()` 函数对所有用户输入的转义 |
